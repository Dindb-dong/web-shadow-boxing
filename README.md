# 웹캠 기반 실시간 포즈 추정 및 궤도 예측을 활용한 AI 쉐도우복싱 파트너

배포 주소: https://web-shadow-boxing.vercel.app/

## 1. 프로젝트 개요

### 1.1. 주제 및 목표

본 프로젝트는 웹캠 기반의 실시간 포즈 추정 기술을 활용하여 사용자의 복싱 동작을 인식하고, 미래의 주먹 궤도(0.3초)와 공격 상태를 예측하여 이에 반응하는 **AI 쉐도우복싱 파트너**를 브라우저 환경에서 구현하는 것을 목표로 한다. 최종 산출물은 단순한 모델 추론 데모를 넘어, 브라우저 단에서 서버리스(Serverless)로 완결되는 인터랙티브 게임 클라이언트로, 사용자의 펀치에 대해 AI 아바타가 위협을 평가하고 회피 및 카운터 공격을 수행하는 완전한 전투 루프를 제공한다.

### 1.2. 핵심 아이디어 및 차별성

- **예측 기반 반응형 시스템:** 단순한 동작 인식 후 반응이 아닌, 최근 시계열 데이터를 기반으로 공격 상태와 미래 궤도를 사전 예측한다.
- **서버리스 런타임 추론:** MediaPipe를 통해 20FPS로 추출된 관절 좌표를 최근 12프레임(0.6초) 윈도우로 구성하고, 이를 브라우저 내장 TypeScript 추론 엔진을 통해 향후 6프레임(0.3초)의 예측값으로 변환한다.
- **학습-서빙 환경의 일관성:** 데이터셋 구축을 위한 오프라인 파이프라인과 브라우저 실시간 파이프라인 간의 정규화, 지수이동평균(EMA) 평활화, 특징 추출 로직을 동일한 규격으로 통일하여 모델의 실제 성능 저하를 방지했다.

## 2. 시스템 아키텍처 및 파이프라인

### 2.1. 오프라인 전처리 파이프라인 (5-Pass)

학습 데이터셋 구축을 위해 아래의 5단계 파이프라인을 설계하였다.

1. **Phase 1 (Pose 추출):** 원본 MP4 영상에서 MediaPipe PoseLandmarker를 이용해 프레임별 3D 좌표를 추출하고, 가시성을 기준으로 유효 프레임을 1차 필터링하여 `.npz` 캐시로 저장한다.
2. **Phase 2 (평활화 및 오프라인 라벨링):** 운동학적 신호에 SMA/EMA를 적용하고, 전체 시퀀스 맥락을 분석하여 공격 구간을 확정하는 오프라인 라벨링을 수행한다.
3. **Phase 3 (타임스탬프 다운샘플링):** 원본 영상의 FPS와 무관하게 20FPS 간격에 가장 가까운 프레임을 추출하여 시계열 해상도를 통일한다.
4. **Phase 4 (슬라이딩 윈도우 생성):** 프레임별 좌표 및 라벨(Idle/Attacking) 데이터를 모델 학습 규격에 맞게 CSV 형태로 저장한다.
5. **Phase 5 (시각화 검증):** 필요시 스켈레톤 및 지표가 오버레이된 MP4를 생성하여 라벨링의 정합성을 검증한다.

### 2.2. 실시간 추론 파이프라인 (게임 클라이언트)

웹캠 입력에서 20FPS 간격으로 MediaPipe 추론을 실행하고, 오프라인과 동일한 전처리를 거쳐 12-step 윈도우 텐서를 구성한다. 이를 경량화된 GRU 모델에 통과시켜 즉각적인 전투 시스템(위협 판정 $\rightarrow$ 회피 $\rightarrow$ 카운터)으로 연결한다.

## 3. 데이터셋 구축 및 라벨링

### 3.1. 관절 추출 및 신뢰도 필터링

전체 동작의 안정성 검증과 노이즈 필터링을 위해 **11개 관절**을 추출하여 가시성(Visibility)을 평가하되, 실제 모델 학습 및 추론 입력으로는 상체의 핵심 움직임을 담는 **6개 관절**만 엄선하여 사용한다.

- **필터링 및 검증용 (11관절):** 코, 양 어깨, 양 팔꿈치, 양 손목, 양 새끼손가락, 양 검지손가락
- **모델 입력용 (6관절):** 양 어깨, 양 팔꿈치, 양 손목
- **필터 기준:** 신체 기준축(코, 양 어깨)은 가시성 0.70 이상, 팔 관절(팔꿈치, 손목)은 0.50 이상을 요구하며, 트래킹 신뢰도는 0.65 이상으로 설정했다.

### 3.2. 오프라인 공격 라벨링 (Heuristic 기반)

오프라인 라벨링 시 모델의 타겟이 되는 `Attacking` 상태는 특정 순간의 이벤트가 아닌 '연속된 타격 구간(Interval)'으로 정의된다. 상체 이동이나 더킹에 의한 오탐을 억제하고, 전방으로 뻗는 실제 펀치의 특징만을 추출하기 위해 5가지 운동학적 신호 복합 조건을 적용했다.

**[표 1: 오프라인 라벨링 핵심 파라미터 및 임계값]**

| **변수명** | **임계값 (Threshold)** | **역할 및 설명** |
| --- | --- | --- |
| `z_vel_thresh` | $\ge 0.040$ | 몸통 이동을 보정한 손목의 Z축(전방) 상대 접근 속도 |
| `ext_thresh` | $\ge 0.012$ | XY 평면상 어깨-손목 거리의 순간 증가량 |
| `WSpd_ok` | $\ge 0.050$ | 몸통 이동을 보정한 손목의 3D 속도 크기 (카메라 앵글 강인성 확보) |
| `FwdRatio_ok` | $\ge 0.40$ | 손목 이동 벡터 중 전방 성분의 비율 (횡이동 및 회수 동작 오탐 방지) |
| `ZPos_ok` | $< -0.08$ | 몸통(어깨 Z 평균) 대비 손목의 전방 돌출 여부 (회수 구간 억제) |
| `BodyV_ok` | $< 0.012$ | 양 어깨의 Y축 평균 속도 (바운스 및 더킹 등 상하 이동 억제) |
- **최종 후보 판정 논리:** `ZPos_ok`와 `FwdRatio_ok` 게이트를 통과한 상태에서, 강한 펀치(`strong_punch`)이거나, 상하 이동이 억제된(`BodyV_ok`) 일반 펀치(`normal_punch`)일 경우 최종 공격 후보로 판정한다.
- **구간화 (Preroll 소급 적용):** 펀치 정점이 확인되면 `attack_preroll_frames`를 소급 적용하여 펀치가 시작되는 준비 동작부터 라벨링을 덮어씌운다.

## 4. 데이터 전처리 및 모델 입력 공간

### 4.1. 공간 정규화 (Normalization)

사용자와 카메라 간의 거리 및 신체 비율 차이를 상쇄하기 위해 양 어깨를 기준으로 3D 좌표를 정규화한다.

- 중심 좌표 ($c_t$): 양 어깨의 중점
- 스케일 ($s_t$): 양 어깨 간의 유클리드 거리 ($s_t = \|L_s - R_s\| + \epsilon$)
- 정규화 수식: $\tilde{p}_t = (p_t - c_t) / s_t$

### 4.2. EMA smoothing 및 Extracting Features

정규화된 좌표에 지수이동평균(EMA)을 적용하여 고주파 노이즈를 제거한다. (학습 데이터 구축 시 $\beta = 0.7$, 실시간 서빙 시 $\beta = 0.5$ 적용). 이후 최근 3개 프레임의 평활화 데이터를 바탕으로 각 관절의 속도와 가속도를 도출한다.

### 4.3. 최종 입력 텐서 구성

- **특징 차원:** 6개 관절 $\times$ (위치 3 + 속도 3 + 가속도 3) = 프레임당 54차원
- **입력 윈도우:** 최근 12프레임(0.6초)을 누적하여 **(Batch, 12, 54)** 형태의 입력 텐서를 구성한다.

## 5. 예측 모델 설계 및 학습

### 5.1. 다중 출력 GRU 아키텍처 (GRUTrajStateModel)

모델은 입력 텐서 `(B, 12, 54)`를 받아 은닉층 크기가 128인 1-Layer GRU를 통과시킨다. 마지막 타임스텝의 은닉 상태(Hidden State)로부터 두 가지 출력을 동시에 생성하는 멀티헤드 구조를 갖는다.

- **State Head:** 현재 프레임의 공격 여부를 이진 분류 (Idle / Attacking).
- **Trajectory Head:** 36차원 벡터를 출력한 뒤 `(B, 2, 6, 3)` 형태로 Reshape 한다. 여기서 `2`는 좌/우 손목 축을 의미하며, `6`은 미래 예측 스텝(0.3초), `3`은 3D 좌표(x, y, z)를 나타낸다.

### 5.2. 2-Phase 학습 전략

상태 분류와 궤도 회귀의 손실 스케일 차이로 인한 학습 불안정을 방지하기 위해 단계적 파인튜닝을 적용했다.

- **Phase 1:** State Head를 동결하고 Trajectory Loss만 사용하여 GRU와 회귀 계층을 선행 학습한다. (Epochs 30, AdamW)
- **Phase 2:** GRU와 Trajectory Head를 동결하고 State Head만 미세 조정한다. (Epochs 80, AdamW)

### 5.3. 손실 함수 (Loss Function)

- **State Loss:** class ****imbalance을 고려한 Cross Entropy를 사용했다. (가중치: Idle 0.523, Attacking 1.477)
- **Trajectory Loss:** 미래 프레임의 오차에 가중치를 부여한 Smooth L1 Loss를 적용했다.

## 6. 실험 및 성능 평가

### 6.1. 데이터셋 통계

총 19,085개의 프레임 샘플을 구축하였으며, 비디오 ID 단위 분리(`by_video_id`)를 통해 학습셋 8,218개, 검증셋 10,867개로 나누었다. 전체 데이터 중 Attacking 라벨 비율은 26%이다.

### 6.2. 정량적 성능 평가 및 해석

시계열 데이터의 특성상 무작위 분리(Random Split) 평가 시 인접 프레임 누출(Data Leakage)로 인해 성능이 과대평가될 위험이 크다. 따라서 본 보고서의 최종 성능 지표는 **완전히 분리된 비디오를 대상으로 한 `by_video_id` 검증 결과를 기준**으로 삼는다.

- **최종 모델 성능 (by_video_id 기준):** 검증 정확도(Val Acc) 83.82%, F1 Score 54.83%
- **임계값 최적화:** 실시간 운영 시 State Head의 공격 판정 임계치(Threshold)를 0.5로 설정했다. 이는 False Positive(허위 회피 동작)를 억제하는 정밀도(Precision)와 실제 펀치에 반응하는 재현율(Recall) 사이의 트레이드오프를 가장 현실적으로 타협한 수치이다.

## 7. 브라우저 기반 게임 클라이언트 구현

### 7.1. TypeScript 기반 서버리스 추론 런타임

무거운 외부 추론 엔진(ONNX Runtime, TensorFlow.js)에 의존하지 않고, PyTorch 체크포인트를 구조화된 JSON 형태로 직렬화하여 클라이언트에 배포했다. 브라우저 내에서 TypeScript로 GRU 게이트 연산을 행렬/벡터 곱으로 직접 구현하여, 완전한 서버리스(Serverless) 초경량 실시간 추론 환경을 달성했다.

### 7.2. 전투 시스템 (위협 $\rightarrow$ 회피 $\rightarrow$ 카운터)

- **위협 판정:** 오프라인 라벨링의 '구간' 개념과 달리, 실시간 게임에서는 모델의 출력이 `Idle`에서 `Attacking`으로 변하는 **상승 엣지(Rising Edge) 이벤트**를 타격 트리거로 정의한다. 트리거 발생 시 예측된 (2, 6, 3) 궤도를 월드 좌표로 변환해 아바타 실루엣과의 교차 여부를 판정한다.
- **회피 및 카운터:** 위협이 성립되면 예측된 궤도의 중심을 피해 확률적으로 회피 방향을 결정한다. 회피 300ms 후 카운터 공격을 발사하며, 유저가 정해진 방어 자세(Tight Guard, Duck, Sway 등)를 취하지 않으면 피격으로 처리된다.

### 7.3. 3D 월드 매핑 및 시각화

정규화 공간에서 예측된 궤적을 Three.js의 월드 좌표계로 투영한다. 단순 역정규화에 그치지 않고, 어깨 앵커를 기준으로 팔의 신장률을 반영하며 와이드 훅(Wide Hook)과 같은 곡선 궤적을 화면 중앙으로 보정하는 로직을 추가하여 시각적 타격감을 극대화했다.

## 8. 결론 및 향후 과제

본 연구는 웹캠으로 추출한 시계열 포즈 데이터를 활용하여 미래 주먹 궤도와 공격 상태를 동시에 예측하는 GRU 기반 딥러닝 파이프라인을 성공적으로 구축하였다. 특히, 정규화 및 평활화 로직을 학습과 서빙 환경에서 엄격하게 일치시키고, 모델 추론을 외부 의존성 없는 클라이언트 내장 스크립트로 구현함으로써 지연 시간(Latency) 없는 서버리스 AI 전투 루프를 완성했다는 데 큰 의의가 있다.

다양한 비디오 환경을 독립적으로 검증(`by_video_id`)하여 모델의 실제 일반화 성능을 확인하였으며, 향후에는 어퍼컷이나 변칙적인 훅 등 다변화된 펀치 궤적에 대한 위협 판정 체감도를 높이고, MediaPipe 모델 로드 최적화를 통해 초기 클라이언트 번들 크기를 경량화하는 고도화 작업이 요구된다.
